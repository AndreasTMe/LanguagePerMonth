### February - Go

The Go implementation explores shared-state concurrency using Go's native primitives: goroutines and channels. It
implements the same producer-consumer challenge as the C# starter, focusing on the "Don't communicate by sharing memory;
share memory by communicating" philosophy.

### Key Technologies

- **Go 1.22+**
- **Channels**: Native language support for typed, thread-safe communication. The implementation uses a buffered channel
  as the shared queue.
- **Goroutines**: Lightweight, user-space threads managed by the Go runtime.
- **sync.WaitGroup**: Used for synchronising the completion of multiple concurrent workers.
- **Context**: Standard library package for handling timeouts, deadlines, and cancellation signals across API boundaries
  and goroutines.

### Implementation Details

#### 1. Configuration (`configuration.go`)

Parses CLI arguments into a `Configuration` struct. Uses standard Go error handling patterns (`if err != nil`).

#### 2. Messaging (`incoming/`)

The `Work` item is encapsulated in a package, mirroring the structure of the C# and Rust versions. It uses Go's approach
to encapsulation (exported vs. unexported fields/methods).

#### 3. Worker Strategy (`main.go` & `worker.go`)

- **Producer**: The main goroutine populates the buffered channel. Once finished, it explicitly closes the channel using
  `close(channel)`, which serves as a signal to all consumers that no more work is coming.
- **Consumer Loop**: Workers use a `for range` or `select` loop over the channel. The `select` block also monitors a
  `context.Context` for graceful shutdown.
- **Orchestration**: A `sync.WaitGroup` ensures the main process doesn't exit until all worker goroutines have finished
  processing their remaining items.

### C# vs. Go

In the context of Distributed Systems, the choice between C# (.NET) and Go often comes down to the trade-off between
framework richness and runtime simplicity.

#### 1. Concurrency Primitives and Runtime

- **C#**: Uses the `Task` abstraction (TPL) which is a promise-based model built on top of a managed thread pool. While
  `System.Threading.Channels` is powerful, it is a library addition rather than a core language keyword.
- **Go**: Concurrency is "baked-in." Goroutines are significantly lighter than C# Tasks in terms of initial stack size.
  For building high-fan-out services (e.g. a reverse proxy or a massive notification dispatcher), Go's scheduler often
  provides more predictable tail latencies under extreme concurrency.

#### 2. Error Handling and Resilience

- **C#**: Relies on Exceptions. In distributed systems, this can lead to "exception storms" if not handled carefully.
  Async/await also introduces the risk of "async all the way," where a single blocking call can starve the thread pool.
- **Go**: Explicit error returns. While often criticised as verbose, from a Distributed Systems perspective, it forces
  the engineer to decide how to handle failure at every hop (retry, circuit break, or propagate). This leads to more
  robust "fail-fast" implementations.

#### 3. Memory Management and GC

- **C#**: The .NET GC is highly sophisticated (Generational, Concurrent, LOH). It is optimised for long-lived
  applications with complex object graphs. However, tuning it for low-latency distributed systems can be complex (e.g.
  choosing between Workstation and Server GC).
- **Go**: The GC is non-generational and concurrent, designed for low latency over high throughput. It aims for
  sub-millisecond pauses. For sidecars and microservices in a K8s environment, Go's smaller binary size and lower, more
  stable memory footprint (RSS) often make it easier to bin-pack on nodes.

#### 4. Communication Patterns

- **C#**: `Channels` in C# are very similar to Go channels but feel more like a "data structure" you pass around.
- **Go**: Channels are a way of life. The `select` statement is a first-class citizen, making it trivial to implement
  complex patterns like fan-in, fan-out, and racing (timeout vs. work). While C#'s `Task.WhenAny` achieves similar
  results, the Go syntax is arguably more readable for orchestration logic.
